# 開発判断ログ方針

## 目的

開発中に人間が行った判断について、判断基準と結果を後から追えるようにする。

ここで扱うのは、ADR にするほど大きくない判断も含む。たとえば、実装範囲、優先順位、採用しなかった案、検証の十分さ、運用上の好みなどを残す。

## 記録するもの

- 判断した日付。
- 判断した人。
- 何を決めたか。
- 判断基準。
- 検討した選択肢。
- 採用した結果。
- 採用しなかった案と理由。
- 影響するファイル、機能、運用。
- 後で見直す条件。

## 記録しないもの

- secrets、credentials、token、customer data、private personal data。
- その場限りで再利用価値がない細かい作業メモ。
- 根拠がない推測を事実のように書いたもの。
- 外部 memory の内容を未検証のまま正本扱いしたもの。

## 置き場所

軽い判断や作業中の判断は、外部 vault の `10-projects/<project>/decisions.md` に書く。

正本 docs に反映すべき判断は、次のどれかへ昇格する。

- architecture、public API、data model、deployment、module boundary に関わる判断: `docs/adr/`
- agent の運用に関わる判断: `docs/ai/`
- build / test / lint に関わる判断: `docs/runbooks/build-test-lint.md`
- 失敗や回避策に関わる判断: `docs/pitfalls/`

## 書き方

- 自然文は日本語で書く。
- 事実、判断、推測を分ける。
- 「なぜそうしたか」を短く残す。
- 未来の agent が同じ論点で迷わない粒度にする。
- 判断が変わった場合は、古い記録を消さずに「変更後の判断」と「変更理由」を追記する。

## 判断ログの形式

```md
## YYYY-MM-DD: <判断タイトル>

- 判断者: <名前または role>
- 状況: <何について決める必要があったか>
- 判断基準:
  - <基準1>
  - <基準2>
- 選択肢:
  - <案A>: <利点 / 欠点>
  - <案B>: <利点 / 欠点>
- 結果: <採用した判断>
- 採用しなかった理由: <必要なら>
- 影響範囲: <files / docs / workflow>
- 見直す条件: <いつ判断を変えるか>
```

## Agent の扱い

agent は、人間が明示した判断や基準を勝手に正本化しない。まず外部 vault の判断ログまたは memory proposal に記録し、正本 docs へ反映する必要がある場合は提案する。
